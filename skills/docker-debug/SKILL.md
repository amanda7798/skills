# Docker 容器调试

从用户提供的参数中解析出容器名和项目目录。参数格式为：`<容器名> <项目目录>`

例如：
- `/docker-debug my_container /workspace/DirectStream`
- `/docker-debug build_env /workspace/linux-ddk`

## 工作约定

- **容器是隔离环境**：容器内的文件系统与宿主机完全隔离。**任何**针对容器内路径（如 `/workspace/`）的操作，无论是 `ls`、`cat`、`find` 还是读取文件，都必须通过 `docker exec` 在容器内执行，绝对不能直接在宿主机上运行。
- **进入容器的唯一方式**：`docker exec <容器名> bash -c "<命令>"`
- **工作目录**：进入容器后默认工作目录为 `/workspace/tmp`，执行命令时以此为基准，除非用户明确指定其他路径
- **项目目录**：用户指定的项目目录是容器内的路径，不是宿主机路径

## 调试原则（重要）

**优先静态分析，最后才执行命令。** 按以下优先级排查：

1. **先看日志**：通过 `docker exec` 查找项目目录内的 log 文件（`*.log`、`build.log`、`stderr`、`CMakeFiles/*.log` 等），从错误信息入手
2. **再读源码/配置**：根据报错，通过 `docker exec` 读取相关代码、CMakeLists.txt、脚本、配置文件
3. **静态推断根因**：结合日志 + 代码，尽量在不重跑构建的情况下分析出问题所在
4. **最后才执行构建/测试命令**：只有静态分析无法确认时，才执行构建或测试；且每次执行前说明目的

> 禁止在未阅读日志/代码的情况下直接重新执行构建/测试命令来"看看报什么错"。

## 调试流程

1. **确认参数**：从用户输入解析容器名和项目目录，如果缺少参数则询问用户
2. **检查容器状态**：运行 `docker ps --filter name=<容器名>` 确认容器正在运行
3. **浏览项目目录**：用 `docker exec <容器名> bash -c "ls <项目目录>"` 了解目录结构
4. **收集现有信息**：通过 `docker exec` 读取项目目录下已有的日志文件、构建产物、错误输出
5. **静态分析**：根据日志报错定位源码、CMakeLists、脚本，通过 `docker exec` 阅读相关文件推断根因
6. **说明方案并询问确认**：在做任何修改或执行任何构建命令之前，必须先向用户说明：
   - **错误原因**：根因是什么，为什么会出现这个问题
   - **修改方案**：要改哪个文件、改什么内容（或要执行什么命令、目的是什么）
   - **询问确认**：明确问用户"是否可以这么操作？"，等待用户明确同意后才继续
7. **按需执行命令**：仅在静态分析不足以确认问题时，才执行最小化的验证命令（检查文件存在、查看环境变量、查看依赖版本等），不重跑完整构建

## 执行命令的方式

**容器内所有操作**（包括查看目录、读文件、运行命令）统一用：
```bash
docker exec <容器名> bash -c "cd /workspace/tmp && <你要执行的命令>"
```

读取容器内文件内容：
```bash
docker exec <容器名> bash -c "cat <文件路径>"
```

如需交互式进入容器，提示用户在终端运行：
```bash
docker exec -it <容器名> bash
```
（Claude 本身无法驱动交互式 TTY，需要用户手动执行）
